Game Server

Spring Boot orchestration layer. WebSocket protocol, state masking (fog of war), lobbies, tournaments, and the thin server pattern.

Architecture

The game server is a thin orchestration layer that wraps the rules engine. It contains zero game logic — all rules enforcement is delegated to the engine.

WebSocket → GameWebSocketHandler → Domain Handlers | GameSession (synchronized) | ActionProcessor (rules engine) | ClientStateTransformer (mask + project) | ServerMessage.StateUpdate | WebSocket

GameSession

The runtime game container. Thread-safe with synchronized(stateLock) on all state mutations.

Key Responsibilities

FeatureDescription
Action ProcessingRoutes GameAction through ActionProcessor, handles Success/PausedForDecision/Failure
Undo SystemCheckpoints at undoable actions (land plays, combat declarations). Invalidated by information-revealing events, stack resolution, turn changes, or opponent actions.
Auto-PassThree modes: AUTO (smart Arena-style), STOPS (stop on opponent stack + combat), FULL CONTROL (never auto-pass)
State MaskingUses ClientStateTransformer to mask state per player + StateDiffCalculator for delta compression
IdempotencylastProcessedMessageId ConcurrentHashMap prevents duplicate action processing
Replay RecordingInitial snapshot + deltas for memory-efficient replays
PersistenceRedis-backed state caching via getStateForPersistence() / restoreFromPersistence()

State Masking (Fog of War)

ClientStateTransformer transforms engine state into client-safe DTOs per player:

ZoneVisibility
LibraryAlways hidden (size only)
HandOnly owner sees cards; opponent sees count
BattlefieldAlways visible (uses projected state)
Graveyard / ExileAlways visible
StackAlways visible

RevealedToComponent overrides hiding for individually revealed cards.

Client DTO

ClientGameState is the sanitized state sent to each player:

data class ClientGameState(
    val viewingPlayerId: EntityId,
    val cards: Map<EntityId, ClientCard>,   // Visible cards only
    val zones: List<ClientZone>,           // Zone info + sizes
    val players: List<ClientPlayer>,       // Life, hand size, etc.
    val currentPhase: Phase,
    val activePlayerId: EntityId,
    val priorityPlayerId: EntityId,
    val combat: ClientCombatState?,
    val gameLog: List<ClientEvent>
)

Tournament System

The TournamentLobby implements a state machine:

WAITING_FOR_PLAYERS | ┌───┴───┐ v v SEALED DRAFT | | v v DECK_BUILDING | v TOURNAMENT_ACTIVE | v TOURNAMENT_COMPLETE

Key Features

  • Dynamic matchmaking: Matches start when both players ready (not waiting for whole round)
  • Per-lobby locks: Prevent race conditions on round advancement
  • Epoch-based validation: Stale ready requests detected via readyEpoch
  • BYE handling: Auto-completes; client auto-readies
  • Spectating: Enabled for all tournament matches

Dev Scenario Controller

REST endpoint (POST /api/dev/scenarios) for E2E testing. Enabled by game.dev-endpoints.enabled=true.

  1. Client sends ScenarioRequest JSON (players, cards in zones, phase)
  2. ScenarioBuilder constructs engine GameState directly
  3. Returns connection tokens for WebSocket
