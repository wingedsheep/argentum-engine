Web Client

React 18.3 browser UI following the dumb terminal pattern. No game rules, no state computation — only intent capture and rendering.

Dumb Terminal Pattern: The server validates all actions, computes all state, and sends complete game state. The client renders what the server sends and captures player clicks.

Data Flow

Server sends stateUpdate (game state, events, legal actions) | v Zustand store updates game state | v React components re-render game board | v User clicks card → Client checks legalActions via hooks | v Client sends submitAction with GameAction | v Server validates, updates state, broadcasts new state

Zustand Store

The store is split into five domain slices:

connectionSlice

StateActions
connectionStatus, playerId, sessionId, aiEnabled connect(playerName), disconnect()

gameplaySlice

StateActions
gameState, legalActions, pendingDecision, mulliganState, fullControl, priorityMode, undoAvailable submitAction(), concede(), requestUndo(), submitDecision(), keepHand(), mulligan(), cyclePriorityMode()

lobbySlice

Tournament lobbies, spectating, standing updates.

draftSlice

Sealed/draft deck building state.

uiSlice

Client-only UI state: targeting mode, combat mode, card selection, animations.

  • Targeting: startTargeting(), addTarget(), confirmTargeting()
  • Combat: startCombat(), toggleAttacker(), assignBlocker()
  • Animations: addDrawAnimation(), addDamageAnimation(), addRevealAnimation()

WebSocket Communication

The GameWebSocket class manages the connection:

  • Auto-reconnect: Exponential backoff (5 attempts, 1s base delay × 2^attempt)
  • State versioning: Detects gaps in message sequence, requests resync
  • Visibility detection: Requests resync when browser tab becomes visible

Hooks

useInteraction

Central card interaction logic. Routes clicks through legal action checks:

processCardClick(cardId): CardClickResult
  → 'noAction'          // Card has no legal actions
  → 'singleAction'      // Auto-execute the one action
  → 'multipleActions'   // Show action menu
  → 'requiresTargeting' // Start targeting flow
  → 'requiresXSelection' // X cost modal
  → 'requiresConvokeSelection'

useTargeting

Manages the targeting flow: startTargeting() → addTarget() → confirmTargeting()

useLegalActions

Queries legal actions for a specific card/entity from the store.

Interaction Flows

Card Click → Action Execution

  1. User clicks card
  2. useInteraction.handleCardClick(cardId) filters legalActions
  3. Result: single action (auto-execute), multiple (show menu), or requires modal
  4. submitAction(action) sends via WebSocket

Targeting Flow

  1. Action requires targets → startTargeting({ validTargets, min, max })
  2. TargetingArrows overlay highlights valid targets
  3. User clicks valid target → addTarget(entityId)
  4. User confirms → confirmTargeting() → submitAction()

Combat Flow

  1. Declare Attackers step → startCombat()
  2. User toggles creatures → toggleAttacker(creatureId)
  3. Declare Blockers step → assignBlocker(blocker, attacker)
  4. User confirms → submits DeclareAttackers/DeclareBlockers

Decision (e.g., Scry)

  1. Server sends pendingDecision (ScryDecision)
  2. DecisionUI routes to CardSelectionDecisionUI
  3. User selects cards
  4. submitDecision(selectedCards) wraps as SubmitDecision action

Selectors

Derived state via Zustand selectors (prefer over computing in components):

const viewingPlayer = useViewingPlayer()
const opponent = useOpponent()
const stackCards = useStackCards()
const battlefieldCards = useBattlefieldCards(playerId)
