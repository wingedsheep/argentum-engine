Rules Engine

The pure functional core. Immutable state, ECS architecture, action processing, Rule 613 state projection, trigger detection, and the continuation system. Zero server dependencies.

GameState

The immutable snapshot of the entire game. Every action produces a new GameState.

PropertyDescription
entitiesMap of EntityId → ComponentContainer (all game objects)
zonesMap of ZoneKey → List<EntityId> (cards per zone)
turnNumberCurrent turn number
activePlayerIdActive player
phase / stepCurrent phase and step
stackSpells and abilities waiting to resolve
priorityPlayerIdWho has priority
pendingDecisionDecision awaiting player input (pauses engine)
continuationStackStack of ContinuationFrames for resuming
floatingEffectsTemporary effects from spells (until end of turn)
delayedTriggersTriggers waiting for specific steps
timestampCounter for Rule 613 timestamp ordering

Lazy projection: projectedState: ProjectedState is cached per instance, applying all Rule 613 layers on-demand.

ActionProcessor

The main entry point. A thin facade over the ActionHandlerRegistry:

  1. Validates basics (game not over, player exists)
  2. Delegates to ActionHandlerRegistry by action type (KClass)
  3. Returns ExecutionResult (new state, events, error/decision)

Component System (ECS)

ComponentContainer is an immutable map keyed by type name:

container.get<CardComponent>()       // Get component or null
container.require<CardComponent>()   // Get or throw
container.has<TappedComponent>()      // Check presence
container.with(component)             // Add/replace (returns new container)
container.without<TappedComponent>()  // Remove (returns new container)

See Components Reference for the complete catalog of component types.

State Projection (Rule 613)

The StateProjector applies continuous effects in layer order to produce ProjectedState:

LayerEffect TypeExample
1Copy effectsClone copying a creature
2Control-changingControl Magic stealing a creature
3Text-changingMagical Hack changing text
4Type-changingMistform becoming a Soldier
5Color-changingShifting Sky changing colors
6Ability add/removeLevitation granting Flying
7aCharacteristic-defining abilitiesTarmogoyf’s */* stats
7bSet P/T to specific valueHumility making everything 1/1
7c+N/+N modificationsGiant Growth giving +3/+3
7dCounter modifications+1/+1 and -1/-1 counters
7eP/T switchingAbout Face switching P/T

Effect Sorting within a Layer

  1. Dependency (trial application): If applying effect A changes the outcome of effect B, then A must be applied first (Rule 613.8)
  2. Timestamp: Fallback ordering if no dependency detected
Critical Rule: Always use projected state for game logic involving battlefield permanents (P/T, controller, types, keywords). Base components hold raw data only. Using base state for filtering misses continuous effects (type-changing, color-changing, etc.).

ProjectedState API

projected.getController(entityId)      // Who controls this?
projected.getPower(entityId)           // Projected power
projected.getToughness(entityId)       // Projected toughness
projected.hasKeyword(entityId, kw)     // Has keyword after effects?
projected.isCreature(entityId)         // Is it a creature after type changes?
projected.hasColor(entityId, color)    // Color after color changes?
projected.isFaceDown(entityId)         // Is it face-down?

Trigger System

TriggerDetector

Observes GameEvents and builds a list of PendingTrigger objects in APNAP order (Active Player, Non-Active Player):

  1. buildTriggerIndex(): Pre-scans battlefield permanents, buckets triggers by event type
  2. Sub-detectors: DeathAndLeaveTriggerDetector, DamageTriggerDetector, AttachmentTriggerDetector
  3. detectTriggers(state, events): For each event, returns matching triggers
  4. detectPhaseStepTriggers(): Phase/step changes (called by PassPriorityHandler)

TriggerProcessor

Puts triggered abilities on the stack:

  • Processes one trigger at a time in APNAP order
  • If trigger has targets: pauses for TargetingDecision, pushes continuation
  • If multiple triggers and first pauses: remaining stored in PendingTriggersContinuation

Continuation System

A serializable alternative to closures. When the engine pauses for player input, it pushes a ContinuationFrame onto the stack:

Engine executing spell | v Needs player input (e.g., "choose cards to scry") | v Push ContinuationFrame onto continuationStack Return PendingDecision to server | v Player responds via SubmitDecision | v ContinuationHandler pops frame, resumes execution

80+ continuation types covering: effect pipeline resumption, targeting, combat damage assignment, modal choices, may effects, clone entering, amplify, storm copies, and more.

Combat System

The CombatManager delegates to three phase managers:

ManagerResponsibility
AttackPhaseManagerValidates attackers (summoning sickness, restrictions, tapped)
BlockPhaseManagerValidates blockers (evasion, flying, unblockable), mandatory blockers, attacker ordering
CombatDamageManagerDeals combat damage (handles first strike + regular steps separately)

Turn Structure

Beginning Phase
  ├─ Untap Step
  ├─ Upkeep Step
  └─ Draw Step

Precombat Main Phase

Combat Phase
  ├─ Begin Combat
  ├─ Declare Attackers
  ├─ Declare Blockers
  ├─ Combat Damage (first strike if needed)
  ├─ Combat Damage (regular)
  └─ End Combat

Postcombat Main Phase

Ending Phase
  ├─ End Step
  └─ Cleanup (damage, discard to hand size)

Mana System

The ManaSolver finds optimal lands to tap for AutoPay:

  1. Pay colored costs first (prefer mono-color sources)
  2. Colorless costs with colorless-only sources
  3. Generic costs with remaining sources

PaymentStrategy: AutoPay (engine taps), FromPool (player floated mana), Explicit (player specifies sources).
