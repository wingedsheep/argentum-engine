Core Design Principles

1. Data-Driven Design

Cards and abilities are purely serializable data structures with zero execution logic. The SDK defines what happens; the engine defines how it happens. 

// This is DATA, not code. It describes behavior without implementing it.
val GiantGrowth = cardDef("Giant Growth") {
    manaCost = "{G}"
    typeLine = "Instant"
    spell {
        effect = Effects.ModifyStats(3, 3, Duration.EndOfTurn)
        target = TargetFilter.Creature
    }
}

Why: State projection can inspect effects without triggering side effects. Cards serialize to JSON for networking. Definitions hot-reload without recompiling the engine.

2. Functional Core with Immutable State

The engine operates as a pure function:

(GameState, GameAction) → ExecutionResult(GameState, List<GameEvent>)

Every action returns a brand-new state. No mutations. This enables time-travel debugging, AI simulations (hold an array of states), and eliminates concurrent modification bugs.

3. Entity-Component-System (ECS)

Instead of OOP inheritance (class Creature : Permanent), every game object is an EntityId with behavioral traits via ComponentContainer:

// A creature that becomes an enchantment? Just swap components.
val entity = container
    .with(CardComponent(name = "Opalescence"))
    .with(ControllerComponent(playerId))
    .with(TappedComponent)
    .without<CreatureTypeComponent>()

Why: In Magic, objects constantly change their fundamental nature. A Land becomes a Creature, gains Flying, then turns face-down. ECS handles this by adding/removing components.

4. AST for Dynamic Values

Numbers are DynamicAmount AST nodes, not lambdas:

// Tarmogoyf: P/T = card types in all graveyards
DynamicAmount.CardTypesInAllGraveyards

// Complex expression: (creatures you control) + 1
DynamicAmount.Add(
    DynamicAmount.Count(Player.You, Zone.BATTLEFIELD, GameObjectFilter.Creature),
    DynamicAmount.Fixed(1)
)

Why: Lambdas can’t be serialized, inspected, or displayed in UI. AST nodes enable lazy evaluation, serialization, and dynamic text generation.

5. Late-Binding Targets

Effects use references like EffectTarget.ContextTarget(0) instead of concrete entity IDs. Targets are chosen at cast time but resolved at resolution time.

6. Composable Filtering

A single GameObjectFilter composes predicates instead of requiring hundreds of specific enums:

// "tapped creature an opponent controls with power 3 or less"
GameObjectFilter.Creature
    .tapped()
    .opponentControls()
    .powerAtMost(3)

7. Base State vs. Projected State

The engine separates Base State (card text + counters) from Projected State (after continuous effects). The StateProjector calculates projected state through Rule 613’s 7 layers:

LayerEffect Type
1Copy effects
2Control-changing effects
3Text-changing effects
4Type-changing effects
5Color-changing effects
6Ability-adding/removing effects
7a–7ePower/toughness modifications (5 sublayers)

8. Reentrant Continuations

When resolution needs player input, the engine doesn’t block. It pushes a serializable ContinuationFrame onto a stack and returns:

CastSpell("Scry 2") → Engine pauses, returns PendingDecision(ScryDecision) → Client shows scry UI, player selects → SubmitDecision(selectedCards) → Engine pops ScrySelectionContinuation, finishes resolution

9. Strategy-Based Registries

Logic is decoupled from the core loop via registries:

  • ActionHandlerRegistry — maps GameAction types to handlers
  • EffectExecutorRegistry — maps Effect types to executors
  • ContinuationResumerRegistry — maps ContinuationFrame types to resumers

Adding a new mechanic means registering a new handler. Zero changes to ActionProcessor.kt.

10. Explicit Event Emission

Every state mutation emits a GameEvent. The TriggerDetector observes events to determine triggered abilities, decoupling side effects from game logic.

Data Flow

Client Click | v WebSocket ClientMessage (JSON) | v GameWebSocketHandler (routing) | v GamePlayHandler (domain logic) | v GameSession (synchronized state container) | v ActionProcessor (rules engine) | v ActionHandler → EffectExecutor | v New GameState + List<GameEvent> | v TriggerDetector (checks events for triggers) | v ClientStateTransformer (mask hidden info, project state) | v ServerMessage.StateUpdate (WebSocket) | v Zustand Store → React Re-render
